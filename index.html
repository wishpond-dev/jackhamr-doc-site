Lifecycle
spec → plan → build → test → review → ship
Every task moves through six phases. The pipeline runs end to end — agents pick up where the last one left off, pausing only at approval gates.

Spec

Write the product spec, mockups, and PDF. Defines what to build before any code is written.

Plan

Implementation planning and architecture decisions. Breaks the spec into developer-sized tasks.

Build

Code writing — one developer task at a time. Each task reads the plan and implements exactly its slice.

Test

E2E browser tests and unit tests. QA agents verify the build against the spec.

Review

Code review plus product review. Checks quality, conventions, and that the result matches the spec.

Ship

Delivery and notify user. Commits land on the branch and the run reports back.

Memory
Inbox → librarian merge → pages
The memory wiki compounds over time. Notes go to an inbox, a periodic librarian pass merges them into pages, and pages are searchable.
Inbox   ►   Librarian   ►   Pages
append-only   dedup & merge   indexed wiki

Inbox

Append-only notes — one file per observation. Nothing edits the inbox in place; it only grows.

Librarian

A periodic merge pass that deduplicates and organizes inbox notes into wiki pages. Single-writer: only the librarian edits pages.

Pages

Synthesized knowledge pages indexed in index.md, searchable via memory_search.

Approval Gates
Human checkpoints in the pipeline
The pipeline pauses at two gates so you can review before work continues. Each gate surfaces a deck with Approve or Request changes.

Gate A — Spec Approval

You review spec.md, mockups, and PDF before the pipeline proceeds.

  • Surfaced via send_deck
  • Approve to continue, or Request changes
  • Blocks planning until resolved

Gate B — Plan Approval

You review architecture decisions and the developer task breakdown before development starts.

  • Surfaced via send_deck
  • Approve to start building, or Request changes
  • Blocks code until resolved

Board
DB-backed task board
The board tracks every task in the run. Tasks are database-backed, hierarchic, and gated by evidence — no marking done without proof.

DB-Backed Tasks

Tasks live in the database, not a file. Tools read and write the board:

  • task_upsert — create or update a task
  • task_mark_done — close a task
  • task_list — read the current plan

Parent-Child Hierarchy

Tasks nest: TASK-NNN parent → TASK-NNN-NNN children.

  • Children must be done before the parent
  • Parent auto-completes when all children close
  • Keeps the run honest about what's finished

Gating Rules

task_mark_done verifies a completed sub-agent run exists.

  • Refuses without evidence
  • PARENT_HAS_PENDING_CHILDREN
  • NO_EVIDENCE_DELEGATION

For a product manager, JackHamr moves the spec-writing bottleneck off your plate without taking you out of the decision. You still decide what to build and why; the spec-builder agent turns that decision into a complete, structured spec — user journey, architecture diagram, acceptance criteria, gap analysis, mockups, and a PDF — and then stops at Gate A so you can review and approve before an implementation plan even exists. Nothing gets built on a spec you haven't seen.

Use Cases

  • Turn a one-paragraph feature idea or a rough PRD into a full spec.md with user stories, acceptance criteria, and visual mockups, ready to hand to engineering
  • Run gap analysis on a feature that's "mostly done" to surface what's actually missing before you promise a launch date
  • Use Gate A as your review checkpoint — approve, or send back with comments, before the implementation-planner agent even starts
  • Triage an incoming bug report into a scoped fix-spec (root cause, affected components, acceptance criteria) without pulling an engineer off their current task just to investigate
  • Track feature work as it moves through spec → plan → build → test → review → ship on the board, so you always know the real status without asking in standup
  • Generate a testing requirements section (unit + E2E test scenarios) alongside the spec, so "done" has a concrete, checkable definition from day one

Example Prompt

"Spec out a CSV export feature for the reporting page — users should be able to export the current filtered view, not just the full table."

JackHamr's conductor spawns the spec-builder agent, which reads your brief, drafts the full spec (including an explicit "filtered view vs. full table" acceptance criterion so that ambiguity can't slip through), generates mockups of the export button and a loading state, and produces a PDF. The pipeline then pauses at Gate A for your review.

Best-Practice Tips

  • Write the initial brief in plain language — the spec-builder is built to ask clarifying questions and document assumptions, not to require a formal PRD upfront
  • Actually read the Gap Analysis section before approving Gate A — it's where hidden scope surfaces
  • Use fix mode (a lighter-weight spec pass with no mockups) for bug reports where you just need root cause + acceptance criteria fast
  • Treat the generated acceptance criteria as the contract for "done" — if QA and engineering disagree later about what shipped, the spec is the source of truth
  • Keep specs in the same repo as the code they describe so history and context stay attached to the feature, not scattered across a separate doc tool

For engineers, JackHamr isn't a single chatbot you prompt for snippets — it's an orchestrated pipeline of named, specialized agents that each own one phase: an implementation-planner maps the codebase and writes a file-level plan, a developer implements it and runs builds until they pass, a qa-tester runs E2E and unit tests against the spec's acceptance criteria, and a code-reviewer audits the resulting diff before it's considered done. A conductor agent owns the whole run and enforces two approval gates — after the spec, and after the plan — so you're reviewing architecture decisions before code exists, not after.

Use Cases

  • Hand an approved spec to the implementation-planner and get back a concrete, file-by-file plan you can review and redirect before any code is written (Gate B)
  • Let the developer agent implement the plan inside a hosted cloud environment — isolated, disposable, and gone the moment the task is done, so it never touches your local machine
  • Get an automatic qa-tester pass (E2E + unit tests against the spec's stated acceptance criteria) and a code-reviewer pass (quality, security, spec compliance) before a diff lands in your queue for final review
  • Package house conventions (naming, error-handling patterns, test structure) into a reusable skill once, instead of repeating the same PR comments across every review
  • Use memory so an agent working on your codebase for the fifth time doesn't need the same "here's how our auth module works" explanation you gave it the first time
  • Build a custom orchestration for a repeatable multi-step chore — a dependency upgrade sweep, a lint-rule migration — that would otherwise eat a day of manual, boring work

Example Prompt

"Implement the CSV export feature per spec.md. Run the full test suite and a code review before marking it done."

The conductor spawns implementation-planner first — it reads spec.md, maps the relevant files, and writes a plan. The pipeline pauses at Gate B for your review. Once approved, developer implements it in an environment, qa-tester runs the spec's E2E scenarios, and code-reviewer audits the diff for quality and spec compliance — you get a reviewed, tested change, not a raw first draft.

Best-Practice Tips

  • Use Gate B deliberately — it's cheaper to redirect a bad architectural call in a plan than in a finished diff
  • Write skills for anything you find yourself explaining to every new agent run — conventions, gotchas, "don't touch this file for X reason" — so it's institutional knowledge, not a repeated prompt
  • Check memory before re-briefing an agent on a codebase quirk it may already know from a previous run
  • Treat the qa-tester and code-reviewer passes as a first review layer, not a replacement for your own — they check against the spec and code quality, not judgment calls only a human on the team would make
  • Use isolated hosted environments for anything experimental or destructive (schema migrations, dependency bumps) so a bad run can't corrupt your local setup

For sales engineers and AEs running technical evaluations, JackHamr's value is turning "can it do X for our stack" into something concrete a prospect's own engineers can review, rather than a verbal promise made in a demo call. Because the platform's spec-builder produces real architecture diagrams and data-flow descriptions from documented capabilities, you can generate a prospect-specific technical-fit spec during an evaluation cycle — and because environments are real hosted sandboxes, you can show a live workflow instead of a screenshot.

Use Cases

  • Generate a technical-fit spec that maps a prospect's stated stack (e.g. GitHub + Jira + Slack) onto JackHamr's board, integrations, and pipeline model, showing exactly how their triage-to-ship workflow would run
  • Spin up a hosted environment to demo a real pipeline run against a sample repo that resembles the prospect's actual codebase, instead of a canned recording
  • Answer an RFP or security questionnaire technical question by pointing to the actual documented capability (e.g. "how are approval gates enforced") rather than guessing
  • Use the board to track POC engineering tasks committed to during a sales cycle, so nothing promised during a call gets forgotten before the follow-up
  • Package a repeatable demo script as a skill so every AE on the team runs the same accurate, up-to-date walkthrough instead of an outdated personal deck

Example Prompt

"This prospect's stack is GitHub, Jira, and Slack. Build a one-pager spec showing how JackHamr's board and integrations would fit their current triage-to-ship workflow."

The spec-builder agent produces a scoped document — an architecture diagram, a component map showing where each of their existing tools plugs in, and a described end-to-end flow — grounded entirely in already-shipped capabilities, ready to walk through live with their engineering stakeholders.

Best-Practice Tips

  • Treat every generated prospect spec as a draft pending Gate A review with an internal engineer before it goes to the customer — don't let a generated document become an accidental commitment
  • Never promise a capability the spec-builder didn't generate from already-documented functionality — if it's not on the site today, it's not something to demo as working
  • Use environments, not screenshots, whenever a prospect's evaluation criteria include "show me it actually running"
  • Loop engineering in via a board task the moment a POC requires anything beyond a standard demo — don't quietly commit engineering time in a sales call
  • Keep the technical-fit spec after the deal closes — it becomes the onboarding document for the customer's own engineering team

For marketing, JackHamr's pipeline-plus-gate model maps cleanly onto content production: draft, review, publish. You can orchestrate a run that produces multiple content assets from one brief, but nothing publishes automatically — the approval gate is the review checkpoint that already fits how marketing content should work, so a generated draft never becomes a published post without a human saying yes.

Use Cases

  • Orchestrate a single run that drafts a blog post, several social copy variants, and an internal one-slide summary from one campaign brief — reviewed together before anything ships
  • Encode your brand voice and style guide once as a skill, so every content-generation run inherits it automatically instead of re-briefing tone each time
  • Use an approval gate before publish as a mandatory human check on anything customer-facing — the pipeline pauses, it doesn't ship on its own
  • Track a content calendar as board cards moving through draft → review → published stages, giving engineering and marketing shared visibility into what's in flight
  • Use memory so campaign context and past decisions ("we tried this angle in Q1, it underperformed") persist across sessions instead of getting re-explained every time
  • Generate mockups for a landing-page variant or feature announcement graphic to pressure-test messaging before a full design pass

Example Prompt

"Draft a launch post for the new Guides section: one blog post, three social variants, and a one-slide internal summary. Pause for my review before anything publishes."

The conductor runs a content-drafting pass across all four assets in one go, using your brand-voice skill for consistent tone, then stops at the approval gate — you review everything together, approve or send back specific pieces, and only approved copy moves forward.

Best-Practice Tips

  • Invest in a brand-voice skill early — it's the single highest-leverage thing that makes every subsequent draft sound like you instead of generic marketing copy
  • Always use an approval gate for anything customer-facing — never let a pipeline auto-publish
  • Log campaign learnings (what worked, what didn't) into memory so future drafts build on real history instead of starting from zero every time
  • Treat every generated draft as a first pass a human edits, not a finished asset — speed comes from skipping the blank page, not skipping review
  • Use the board to keep marketing's in-flight content visible to engineering, especially for anything referencing a feature that isn't shipped yet

For finance, JackHamr is less something you use day-to-day and more something you evaluate and model the ROI of adopting elsewhere in the org. Because the spec-builder documents its assumptions explicitly (as an Acceptance Criteria and Open Questions table, not a hidden formula), a finance stakeholder can request a cost-savings or time-savings model as a structured spec and actually audit the reasoning behind the number, rather than taking someone else's spreadsheet on faith.

Use Cases

  • Request a spec that models estimated engineering-hours saved if a team adopts a specific JackHamr capability (e.g. automated code review on all PRs over a size threshold), with assumptions stated explicitly and open questions flagged rather than hidden
  • Use the acceptance criteria section of any such spec as the basis for measuring realized savings after adoption, instead of a one-time hand-wavy estimate
  • Use approval gates as a structural analog for a spend sign-off checkpoint on anything tied to environment usage scaling
  • Build a skill for a recurring finance process (e.g. a monthly close checklist walkthrough) so the documentation of that process — not the actual financial data — stays consistent across cycles
  • Keep a memory page of finance-approved modeling assumptions (e.g. "assume $X/engineer-hour") so every future model an agent builds is consistent with the last one, instead of re-litigating assumptions each time

Example Prompt

"Build a one-page spec estimating cost savings if engineering moves code review to JackHamr's code-reviewer agent for all PRs over 200 lines, using our standard assumptions."

The spec-builder produces a spec with an explicit assumptions table, a described methodology, and acceptance criteria defining exactly how "savings realized" would be measured post-adoption — a document you can hand to a stakeholder and defend, not a single unexplained number.

Best-Practice Tips

  • Treat every generated model as a draft to verify manually — confirm the stated assumptions match reality before using the number in a decision
  • Require a human sign-off (mirroring an approval gate) on anything with a dollar commitment attached, rather than acting on a generated estimate directly
  • Maintain a memory page of standing, finance-approved modeling assumptions so future estimates stay internally consistent quarter over quarter
  • Don't route sensitive, unreleased financial data through a hosted environment without first confirming your organization's data-handling policy covers it — this platform has no dedicated finance-data controls beyond your own environment configuration
  • Use the board to track recurring, process-style finance automation tasks (e.g. close-checklist documentation runs) the same way engineering tracks feature work

For legal and compliance reviewers, the most relevant part of JackHamr isn't a feature you'd use directly — it's the fact that every spec the platform produces already includes an architecture diagram and, where relevant, a data schema section describing exactly what data moves where. That means you can require a "data-flow spec" as a standing precondition before engineering builds any new integration that touches customer or employee data, and review it before code exists rather than after.

Use Cases

  • Require a spec — specifically its Architecture Diagram and Data Schema sections — for any new third-party integration before implementation begins, so data-flow questions get raised at design time, not during a post-launch audit
  • Use the platform's approval gates as the technical analog of a legal sign-off checkpoint: nothing proceeds past Gate A (spec approved) or Gate B (plan approved) without an explicit yes
  • Read the Out of Scope section of any spec closely — it's the most reliable place to catch quietly-expanding scope before it becomes a compliance gap
  • Track legal-review tasks on the same board engineering uses, so a feature can't silently ship without a recorded sign-off
  • Maintain a memory page of standing legal/compliance requirements so future specs get written against them automatically instead of relying on every engineer remembering the current policy

Example Prompt

"Before we build the Slack integration, generate a spec describing exactly what customer data would flow through it, and where it would be stored."

The spec-builder produces a spec whose Architecture Diagram and Data Schema sections lay out the full data path — what's read from Slack, what's persisted, what's transient — reviewable before any implementation work starts, rather than reconstructed after the fact from code.

Best-Practice Tips

  • Make the Data Schema and Architecture Diagram sections a mandatory Gate A review item for any spec touching customer or employee data — don't approve without reading them
  • Never skip an approval gate for compliance-sensitive work, even under deadline pressure — it's the one built-in checkpoint designed for exactly this
  • Keep a standing memory page of current legal/compliance requirements so they get incorporated into specs by default rather than caught late
  • Require sign-off to be recorded in the relevant board task's notes, not just given verbally, for auditability
  • Review generated content (in Guides, Marketing copy, etc.) for accuracy claims about the platform before it's published externally — speculative content should never overstate a capability that isn't actually shipped

For operations, JackHamr's combination of custom orchestrations, hosted environments, and the board is well suited to recurring internal tooling work — the kind of task that's tedious enough to automate but risky enough that you want it running somewhere isolated, with its results tracked, not silently succeeding or failing in a cron job nobody watches.

Use Cases

  • Build a custom orchestration for a recurring task — a weekly access review, a dependency-freshness audit, a log-rotation check — instead of a one-off script only one person remembers how to run
  • Run ops scripts inside a hosted, disposable environment rather than on a local machine or a shared box, so a bad run can't leave lingering state behind
  • File findings from an automated audit run directly as board tasks, so "the weekly check found 3 stale API keys" becomes a tracked item, not a Slack message that gets lost
  • Write incident/runbook steps as a skill so any agent instance — not just whichever engineer is on call — responds to a known incident type the same, consistent way
  • Track infrastructure and ops tickets on the same board engineering uses, giving ops the same visibility into in-flight work that engineering already has

Example Prompt

"Every Monday, run a spec-only audit pass that checks our integrations list for stale API keys and files a board task for anything found."

A scheduled orchestration spawns an agent that runs the audit, produces a findings report, and creates a board task for each stale key found — a recurring check that used to require someone remembering to do it manually now runs unattended and surfaces results where the team already looks.

Best-Practice Tips

  • Only automate what's safe to run unattended — route anything with production-impacting side effects through an approval gate rather than letting a scheduled pipeline run it directly
  • Use isolated hosted environments for any script that touches infrastructure state, so a failed or buggy run can't corrupt a shared machine
  • Centralize runbooks as skills so incident response is consistent regardless of who (or which agent) is on call
  • Give ops the same board visibility engineering has — don't let ops tickets live in a separate, less-visible system
  • Periodically review environment usage as part of routine ops housekeeping, the same way you'd review any other recurring infrastructure cost

For HR, the safest and most useful application of JackHamr is internal documentation and onboarding content generation — the same spec-builder pattern used for product docs works just as well for a new-hire checklist or an internal policy FAQ. It's worth being explicit about the boundary: JackHamr has no backend or database of its own for this feature set, so anything genuinely sensitive (personnel records, case data, compensation details) belongs in your existing HR systems of record, not in agent memory or a generated doc.

Use Cases

  • Draft a structured onboarding checklist for a new hire on a specific team, including which internal skills, docs, and board conventions they should learn first
  • Build a skill that encodes commonly-asked, low-stakes policy questions (e.g. expense process, standard leave categories) so employees get consistent, accurate answers instead of tribal knowledge
  • Track onboarding task completion per new hire on the board, the same way engineering tracks feature work, for shared visibility into ramp progress
  • Draft internal training material or an FAQ page using the same spec-builder pattern used elsewhere on this site — structured, reviewed, versioned
  • Keep a memory page of current internal policies so an "onboarding assistant" skill stays accurate as policies change, rather than drifting out of date silently

Example Prompt

"Draft a one-week onboarding checklist for a new engineer joining the platform team, including which JackHamr skills and docs they should read first."

The spec-builder produces a structured, day-by-day checklist — internal documentation, not a formal HR record — that a new hire's manager can hand over and adapt.

Best-Practice Tips

  • Keep JackHamr-generated HR content strictly advisory — final policy decisions and any sensitive employee data stay in your actual HR system of record, never in agent memory or a generated document
  • Use skills for FAQ-style, low-stakes content only — never for confidential personnel matters or case-specific guidance
  • Have a human review any generated onboarding or policy material before it's rolled out to employees
  • Use the board to track onboarding task completion, not to store any sensitive HR case data
  • Treat memory pages built for this use case as internal team knowledge (e.g. "how our team uses JackHamr"), not as a substitute for an employee record system
