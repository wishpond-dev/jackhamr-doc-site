Lifecycle
spec → plan → build → test → review → ship
Every task moves through six phases. The pipeline runs end to end — agents pick up where the last one left off, pausing only at approval gates.

Spec

Write the product spec, mockups, and PDF. Defines what to build before any code is written.

Plan

Implementation planning and architecture decisions. Breaks the spec into developer-sized tasks.

Build

Code writing — one developer task at a time. Each task reads the plan and implements exactly its slice.

Test

E2E browser tests and unit tests. QA agents verify the build against the spec.

Review

Code review plus product review. Checks quality, conventions, and that the result matches the spec.

Ship

Delivery and notify user. Commits land on the branch and the run reports back.

Agents

Soul Prompts

A short directive that shapes how an agent communicates and works.

Quietly competent, direct,

without drama.

Identity Traits

Traits that shape every message the agent sends:

  • Tone — formal, casual, dry, warm
  • Style — terse vs. verbose, technical vs. plain
  • Approach — cautious vs. bold, incremental vs. big-swing
  • Personality — how it handles mistakes and pushback

Named Agents

Per-agent customization and reuse across the org:

  • Per-agent customization of prompts and traits
  • Org-wide sharing — teammates use the same agent
  • Clone and reuse for new workflows
  • Distinct roles: Orchestrator, Developer, QA, Reviewer

Skills
Reusable, shareable agent skills
Skills are packaged instructions an agent loads on demand. Create them, share them across the org, and keep context lean with progressive disclosure.

Custom Skills

Create and share agent skills for any workflow. A skill bundles instructions, examples, and conventions into one reusable unit.

Progressive Disclosure

Only the skill name and description load in context. Full instructions load on demand via skill_read — keeps prompts lean.

Always-On Skills

Some skills are always active — their instructions apply every turn without an explicit load. Used for cross-cutting habits like memory.

Memory
Inbox → librarian merge → pages
The memory wiki compounds over time. Notes go to an inbox, a periodic librarian pass merges them into pages, and pages are searchable.
Inbox   ►   Librarian   ►   Pages
append-only   dedup & merge   indexed wiki

Inbox

Append-only notes — one file per observation. Nothing edits the inbox in place; it only grows.

Librarian

A periodic merge pass that deduplicates and organizes inbox notes into wiki pages. Single-writer: only the librarian edits pages.

Pages

Synthesized knowledge pages indexed in index.md, searchable via memory_search.

Approval Gates
Human checkpoints in the pipeline
The pipeline pauses at two gates so you can review before work continues. Each gate surfaces a deck with Approve or Request changes.

Gate A — Spec Approval

You review spec.md, mockups, and PDF before the pipeline proceeds.

  • Surfaced via send_deck
  • Approve to continue, or Request changes
  • Blocks planning until resolved

Gate B — Plan Approval

You review architecture decisions and the developer task breakdown before development starts.

  • Surfaced via send_deck
  • Approve to start building, or Request changes
  • Blocks code until resolved
Environments
Docker-based cloud containers
Each run provisions a cloud container with everything the agent needs — editor, terminal, browser, Docker, and git — preconfigured and ready.

VS Code

Editor built in. Extensions and settings ship with the container, so the agent starts in a real IDE.

Terminal

Full SSH access. Agents run real shell commands, scripts, and build tools just like a dev would.

Browser

Built-in for testing and previewing. QA agents drive real E2E browser tests against the running app.

Docker

docker-compose support out of the box. Spin up databases, queues, and services alongside the app.

Git

One-click clone from any repository, with auto-sync. Branches stay current as agents commit.

Board
DB-backed task board
The board tracks every task in the run. Tasks are database-backed, hierarchic, and gated by evidence — no marking done without proof.

DB-Backed Tasks

Tasks live in the database, not a file. Tools read and write the board:

  • task_upsert — create or update a task
  • task_mark_done — close a task
  • task_list — read the current plan

Parent-Child Hierarchy

Tasks nest: TASK-NNN parent → TASK-NNN-NNN children.

  • Children must be done before the parent
  • Parent auto-completes when all children close
  • Keeps the run honest about what's finished

Gating Rules

task_mark_done verifies a completed sub-agent run exists.

  • Refuses without evidence
  • PARENT_HAS_PENDING_CHILDREN
  • NO_EVIDENCE_DELEGATION
